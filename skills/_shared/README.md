# Shared review-doc shell

Every skill that produces a reviewable HTML artifact (brainstorming, spec-writing, impl-plan-writing, test-review) uses the same bake-time pattern:

- `<slug>.content.json` — the source of truth (the skill reads/writes this on revisions).
- `<slug>.html` — generated by `gen-artifact.mjs`; a single self-contained file with CSS, JS, and all content inlined. Opens directly in any browser.
- `<slug>.comments.json` — inline-comment state, managed by the review server.

On every revision: edit the JSON, re-run the generator, overwrite the HTML. Token-wise the skill only ever touches the JSON.

## Layout

```
skills/_shared/
  assets/
    shell.css            # all common styles, inlined as <style>
    shell.js             # mermaid + hljs + TOC + tag pills + comment system, inlined as <script type="module">
    shell.tmpl.html      # template with {{...}} placeholders
  scripts/
    gen-artifact.mjs     # the bake script
  README.md              # this file
skills/<skill>/
  shell.config.json      # per-skill config: title, sections, footer "Next" chip
```

There's no per-skill `assets/shell.html` — that intermediate file was needed only when the shell was fetched at runtime. The generator now produces the final artifact HTML directly from the template + config + content JSON.

## Running the generator

```bash
node ~/.claude/skills/_shared/scripts/gen-artifact.mjs <skill> <content-json-path>
```

For example:

```bash
node ~/.claude/skills/_shared/scripts/gen-artifact.mjs brainstorming \
  docs/brainstorming/2026-05-14-feature.content.json
```

Output: `docs/brainstorming/2026-05-14-feature.html` — single self-contained file.

## Adding a new skill

1. Write `skills/<skill>/shell.config.json`. Schema:

   | Key | Type | Notes |
   |---|---|---|
   | `title` | string | Used as the `<title>` suffix ("Engineering spec review", etc.) |
   | `metaLabel` | string | Shown in the `.meta` strip above the H1 (e.g., "Engineering spec review · 2026-05-14") |
   | `next` | string \| empty | Footer "Next: …" chip; empty hides it |
   | `showGoal` | bool (default true) | Whether to emit the `<p class="goal">` line under the H1 |
   | `showChangelog` | bool (default true) | Whether to emit the collapsible changelog block |
   | `sections[]` | array | Section list — see below |

   Each section:

   | Key | Notes |
   |---|---|
   | `id` | Anchor id (kebab-case) |
   | `heading` | H2 text — HTML-escaped already (use `&amp;` for `&`) |
   | `container` | `div` or `ul` (use `ul` when the JSON value is bare `<li>` elements) |
   | `containerClass` | optional class — e.g., `risks`, `ac-list` |
   | `bind` | Content JSON key to inject as the section's inner HTML |

2. The skill's SKILL.md documents the runtime workflow:
   - Write `<slug>.content.json` matching the section binds.
   - Run `gen-artifact.mjs <skill> <content-json-path>` to produce the HTML.

## Editing shared CSS or JS

`_shared/assets/shell.css` and `_shared/assets/shell.js` are inlined at bake time. Edit them, then **re-run the generator on any artifacts you want updated** — old artifacts retain whatever CSS/JS was inlined when they were last baked.

For a one-off skill-specific CSS extension, add it to `shell.css` guarded by a class your skill applies. There's no per-skill override file in the current design; if one's needed, add an `extras` config and an `{{EXTRA_INLINE_STYLE}}` placeholder.

## Mermaid + highlight.js are CDN-loaded

These two libraries are too large to inline (~600KB combined). The artifact HTML loads them from jsDelivr at view time. Offline viewing renders text only — no diagrams, no syntax highlighting. That trade-off is intentional.

## Skill-agnostic JS

`_shared/shell.js` does NOT hardcode any skill name. It:

- Derives the comments URL from `location.pathname`
- Runs mermaid, hljs, TOC build, tag pills, and the comment system over whatever's already in the DOM
- Has no fetch for content — the content is baked in at gen time

If a skill needs runtime behavior beyond this, add it to `_shared/shell.js` (gated by a feature check) rather than per-skill — the simplicity is worth more than the customization.
